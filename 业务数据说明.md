# 艺术培训机构排课系统 - 业务数据说明

## 2.4.1 数据项

数据项是不可再分的最小数据单元。

---

## 2.4.2 数据结构

由若干个数据项组成，反映数据之间的组合关系。

---

### 表1 用户信息数据结构

| 名称 | 描述 |
|------|------|
| 数据结构编号 | B01 |
| 数据结构名称 | 用户信息 |
| 简述 | 系统登录账户及角色信息 |
| 数据结构组成 | 用户ID + 用户名 + 密码(加密) + 显示名 + 角色 + 关联教师ID + 关联学生ID + 创建时间 |

**处理逻辑：** 存储在 `users` 表。密码使用 BCrypt 加密，登录后由后端签发 JWT Token（有效期 24 小时），前端存入 localStorage 并在每次请求头中携带 `Authorization: Bearer <token>`。角色分三类：`admin`（管理员）、`teacher`（教师）、`student`（学生），决定前端菜单显示范围及后端接口访问权限。教师账号通过 `teacher_id` 关联 teachers 表，学生账号通过 `student_id` 关联 students 表。

---

### 表2 教师信息数据结构

| 名称 | 描述 |
|------|------|
| 数据结构编号 | B02 |
| 数据结构名称 | 教师信息 |
| 简述 | 艺术培训机构的任课教师档案 |
| 数据结构组成 | 教师ID + 姓名 + 联系电话 + 专业方向 + 入职日期 + 可授课程列表 + 创建时间 + 更新时间 + 删除标记 |

**处理逻辑：** 存储在 `teachers` 表，使用逻辑删除（`deleted` 字段，0=正常，1=已删除）。可授课程通过 `teacher_courses` 关联表存储，一名教师可对应多个课程名称。专业方向为文本字段，用于描述教师主攻方向（如钢琴、芭蕾舞、素描等）。仅管理员可增删改查教师信息；教师课表查询时通过 `teacher_id` 筛选该教师名下的所有排课记录。

### 表3 学生信息数据结构

| 名称 | 描述 |
|------|------|
| 数据结构编号 | B03 |
| 数据结构名称 | 学生信息 |
| 简述 | 报名学员的个人及家长联系信息 |
| 数据结构组成 | 学生ID + 姓名 + 年龄 + 性别 + 家长姓名 + 联系电话 + 入学日期 + 报名课程列表 + 创建时间 + 更新时间 + 删除标记 |

**处理逻辑：** 存储在 `students` 表，使用逻辑删除。报名课程通过 `student_courses` 关联表存储，记录学生所学课程名称。性别取值：男/女。仅管理员可增删改查学生信息。学生课表查询时，通过 `class_students` 关联表查找该学生所在班级，再联查该班级的所有排课记录。

### 表4 课程信息数据结构

| 名称 | 描述 |
|------|------|
| 数据结构编号 | B04 |
| 数据结构名称 | 课程信息 |
| 简述 | 培训机构开设的艺术课程目录 |
| 数据结构组成 | 课程ID + 课程名称 + 课程类别 + 授课类型 + 课时时长 + 适合年龄段 + 单节价格 + 课程简介 + 创建时间 + 更新时间 + 删除标记 |

**处理逻辑：** 存储在 `courses` 表，使用逻辑删除。课程类别固定为：音乐、舞蹈、美术三类，用于前端分类筛选和课程卡片颜色区分（音乐-紫色、舞蹈-粉色、美术-橙色）。授课类型分为：小组课（多名学生同时上课）和一对一（单独辅导）。课时时长单位为分钟，范围 45～180 分钟。仅管理员可增删改查课程信息；排课时通过课程ID关联，课程名称冗余存储在 `classes.course_name` 用于快速展示。

|------|------|
| 数据结构编号 | B05 |
| 数据结构名称 | 教室信息 |
| 简述 | 培训机构的场地及设备信息 |
| 数据结构组成 | 教室ID + 教室名称 + 教室类型 + 容纳人数 + 所在楼层 + 使用状态 + 备注 + 设备列表 + 创建时间 + 更新时间 + 删除标记 |

**处理逻辑：** 存储在 `classrooms` 表，使用逻辑删除。教室类型分为：音乐教室、舞蹈教室、美术教室、多功能教室。使用状态取值：空闲/使用中，由管理员手动维护。教室设备通过 `classroom_equipment` 一对多关联表存储，每条记录为一件设备名称（如钢琴、镜子墙、画架等）。教室课表查询时通过 `classroom_id` 筛选该教室内所有排课记录。仅管理员可增删改查教室信息。

### 表6 班级信息数据结构

| 名称 | 描述 |
|------|------|
| 数据结构编号 | B06 |
| 数据结构名称 | 班级信息 |
| 简述 | 将学生、课程、教师组合成可排课的教学单元 |
| 数据结构组成 | 班级ID + 班级名称 + 课程类别 + 课程名称 + 负责教师ID + 班级容量 + 上课时间描述 + 开班日期 + 学生列表 + 创建时间 + 更新时间 + 删除标记 |

**处理逻辑：** 存储在 `classes` 表（因 `class` 为 Java 关键字，实体类命名为 `Clazz`），使用逻辑删除。班级是排课的核心调度单元，一次排课记录（schedules）必须关联一个班级。班级与学生的多对多关系通过 `class_students` 中间表维护。`schedule` 字段为人工填写的文字描述（如"周一、周三 09:00-10:00"），仅作展示用，不参与自动排课逻辑。教师只能查看和申请调整自己名下班级的课程。

---

### 表7 排课记录数据结构

| 名称 | 描述 |
|------|------|
| 数据结构编号 | B07 |
| 数据结构名称 | 排课记录 |
| 简述 | 某班级在某日期某时段的一次具体上课安排 |
| 数据结构组成 | 排课ID + 班级ID + 课程ID + 教师ID + 教室ID + 上课日期 + 星期数 + 开始时间 + 结束时间 + 状态 + 创建时间 + 更新时间 |

**处理逻辑：** 存储在 `schedules` 表。每条记录代表一节具体的课，精确到日期和时段。星期数（`day_of_week`）取值 1-7（1=周一，7=周日），与日期冗余存储便于按周查询。状态取值：`scheduled`（已排课）、`completed`（已完成）、`cancelled`（已取消）、`adjusted`（已调课）。

支持两种创建方式：
- **手动排课**：管理员指定班级、课程、教室、时间、重复类型（单次/按周重复N次），后端循环生成对应条数记录；
- **智能排课**：管理员设置时间范围和约束条件，后端自动分配教室和时段（功能扩展预留）。

课表查询支持四个维度：按周（返回指定周内全部排课）、按教师、按学生（通过班级关联）、按教室，均以周为单位返回数据。

---

### 表8 调课申请数据结构

| 名称 | 描述 |
|------|------|
| 数据结构编号 | B08 |
| 数据结构名称 | 调课申请 |
| 简述 | 教师发起的课程时间调整申请及审批记录 |
| 数据结构组成 | 申请ID + 申请人用户ID + 班级ID + 关联排课ID + 原上课时间描述 + 新星期 + 新开始时间 + 新结束时间 + 申请原因 + 审批状态 + 拒绝原因 + 申请时间 + 审批时间 + 审批人用户ID |

**处理逻辑：** 存储在 `adjust_requests` 表。审批状态三态流转：`待审核` → `已通过` / `已拒绝`，仅管理员可执行审批操作。教师只能查看和撤销自己提交的申请，且只能撤销处于"待审核"状态的申请；管理员可查看所有人的申请记录。申请中的"新时间"字段（`new_day_of_week`、`new_start_time`、`new_end_time`）为可选项，允许只填写原因而不指定新时段。

仪表盘的"待办事项"模块自动拉取状态为"待审核"的申请列表：管理员看到所有待审核申请，教师只看到自己提交的待审核申请，学生端不显示此模块。点击"处理"按钮直接跳转至调课管理页面。

# 艺术培训机构排课系统业务流程说明

## 一、用户登录流程

```plain
开始
  ↓
用户打开浏览器访问系统
  ↓
判断：localStorage 中是否存在 token？
  ├─ 是 → 路由守卫放行 → 跳转仪表盘
  └─ 否 → 强制跳转登录页
           ↓
       用户输入用户名和密码
           ↓
       前端校验：字段是否为空？
           ├─ 否 → 显示提示 → 停留登录页
           └─ 是 → 提交到后端 POST /api/auth/login
                    ↓
                 后端查询 users 表
                    ↓
                 BCrypt 验证密码
                    ↓
                 判断：验证通过？
                    ├─ 否 → 返回 401 → 显示"用户名或密码错误"
                    └─ 是 → 生成 JWT Token（含 userId / username / role，有效期 24 小时）
                             ↓
                          返回 { token, userInfo }
                             ↓
                          前端将 token 存入 localStorage
                          前端将 userInfo 存入 Pinia Store 并持久化
                             ↓
                          跳转仪表盘
```

## 二、仪表盘数据加载流程

```plain
用户进入仪表盘页面
  ↓
前端发起请求：GET /api/dashboard
  ↓
后端 JwtAuthFilter 解析 Token → 注入当前用户到 SecurityContext
  ↓
DashboardController 读取当前用户 role
  ↓
判断角色：
  ├─ admin（管理员）→ 查询：
  │     教师总数、学生总数、本周排课数、待审核调课申请数
  │
  ├─ teacher（教师）→ 查询：
  │     本人本周课时数、所带班级数、本人待审核申请数
  │
  └─ student（学生）→ 查询：
        本人本周课程数、所在班级数
  ↓
前端渲染统计卡片
  ↓
前端发起请求：GET /api/adjustments?status=待审核&size=100
  ↓
后端按角色过滤：
  ├─ admin → 返回全部待审核申请
  └─ teacher → 返回本人提交的待审核申请
  ↓
前端渲染"待办事项"列表，每条显示申请人和班级名
  ↓
用户点击"处理"按钮 → 跳转至调课管理页
```

## 三、排课管理流程

### 3.1 手动排课流程

```plain
管理员进入"手动排课"页面
  ↓
页面加载时并发请求：
  ├─ GET /api/classes?size=100 → 获取班级列表
  ├─ GET /api/courses/options  → 获取课程下拉列表
  └─ GET /api/classrooms       → 获取教室列表
  ↓
管理员依次填写：
  1. 选择班级
  2. 选择课程
  3. 选择教室
  4. 选择星期（周一至周日）
  5. 填写开始日期
  6. 填写上课时间段（开始时间、结束时间）
  7. 选择重复类型：
     ├─ 单次 → 只生成 1 条记录
     └─ 按周重复 → 填写重复周数 N → 生成 N 条记录
  ↓
管理员点击"确认排课"
  ↓
前端校验：必填字段是否完整？
  ├─ 否 → 显示提示 → 停留当前页
  └─ 是 → 提交 POST /api/schedules/manual
           ↓
        后端校验班级是否存在
           ↓
        从 startDate 起找到第一个符合 dayOfWeek 的日期
           ↓
        循环按周生成 N 条 schedules 记录，每条调用 selectByIdWithExtra 填充关联名称
           ↓
        返回已创建的排课列表
           ↓
        前端显示成功提示，展示新建课表
```

### 3.2 课表查看流程

```plain
用户进入课表页（周课表 / 教师课表 / 学生课表 / 教室课表）
  ↓
判断课表类型：
  ├─ 周课表 → GET /api/schedules/week?date=YYYY-MM-DD
  │           后端对齐到当周周一，查询周一至周日全部排课
  │
  ├─ 教师课表 → 判断角色：
  │            ├─ admin → 显示教师下拉选择器，切换教师触发重新请求
  │            └─ teacher → 自动使用 userInfo.teacherId
  │            → GET /api/schedules/teacher/{teacherId}?date=
  │
  ├─ 学生课表 → 判断角色：
  │            ├─ admin → 显示学生下拉选择器，切换学生触发重新请求
  │            └─ student → 自动使用 userInfo.studentId
  │            → GET /api/schedules/student/{studentId}?date=
  │            后端通过 class_students 关联表查找学生所在班级，再联查排课记录
  │
  └─ 教室课表 → 选择教室 → GET /api/schedules/classroom/{classroomId}?date=
  ↓
后端返回当周排课列表（含 courseName / teacherName / classroomName 等关联字段）
  ↓
前端按 dayOfWeek 和 startTime 将课程填入 CSS Grid 网格单元格
  ↓
课程卡片按 category 着色：音乐（紫）/ 舞蹈（粉）/ 美术（橙）
  ↓
用户点击课程卡片 → 弹出详情 Modal（课程名、教师、教室、时间、学生列表）
  ↓
用户点击日期选择器切换周 → 重新发起请求
  ↓
用户点击"今天" → 重置到当前周
```

## 四、调课申请流程

### 4.1 教师发起申请

```plain
教师进入"调课申请"页面
  ↓
GET /api/adjustments → 仅返回本人提交的申请列表（后端自动过滤）
  ↓
教师点击"新建申请"
  ↓
GET /api/classes?teacherId=本人ID → 加载本人名下班级下拉列表
  ↓
教师填写申请表单：
  1. 选择班级
  2. 填写原上课时间描述
  3. 填写申请原因
  4. 选填：新星期、新开始时间、新结束时间
  ↓
教师提交 POST /api/adjustments
  ↓
后端校验：
  ├─ 班级是否存在？
  └─ 该班级是否属于当前教师？
     ├─ 否 → 返回错误"只能申请自己班级的调课"
     └─ 是 → 写入 adjust_requests 表，状态置为"待审核"
              ↓
           返回申请记录（含申请人姓名、班级名等）
              ↓
           前端刷新列表，显示"待审核"状态标签
```

### 4.2 管理员审批流程

```plain
管理员进入"调课管理"页面
  ↓
GET /api/adjustments → 返回全部申请列表（含申请人姓名、班级名）
  ↓
管理员查看某条"待审核"申请
  ↓
管理员选择操作：
  ├─ 审批通过 → 点击"通过"按钮
  │             ↓
  │          PUT /api/adjustments/{id}/approve
  │             ↓
  │          后端将状态更新为"已通过"，记录审批时间和审批人
  │             ↓
  │          前端刷新，状态标签变为"已通过"
  │
  └─ 审批拒绝 → 点击"拒绝"按钮 → 弹出拒绝原因输入框
                ↓
             填写拒绝原因 → 确认提交
                ↓
             PUT /api/adjustments/{id}/reject { rejectReason }
                ↓
             后端将状态更新为"已拒绝"，记录拒绝原因、审批时间和审批人
                ↓
             前端刷新，状态标签变为"已拒绝"，显示拒绝原因
```

### 4.3 撤销申请流程

```plain
教师查看自己处于"待审核"状态的申请
  ↓
点击"撤销"按钮
  ↓
DELETE /api/adjustments/{id}
  ↓
后端校验：
  ├─ 该申请是否属于当前用户？
  └─ 状态是否为"待审核"？
     ├─ 否 → 返回错误"只能撤销自己的待审核申请"
     └─ 是 → 物理删除该条记录
              ↓
           前端从列表移除该条目
```

## 五、基础数据管理流程（以教师管理为例）

```plain
管理员进入"教师管理"页面
  ↓
GET /api/teachers?page=1&size=10 → 返回分页教师列表
  ↓
管理员操作：
  ├─ 搜索 → 输入关键字 → 前端触发 GET /api/teachers?search=关键字
  │
  ├─ 新增教师 → 点击"新增"按钮 → 弹出表单 Modal
  │             填写：姓名、电话、专业方向、可授课程（多选）
  │             ↓
  │          提交 POST /api/teachers
  │             ↓
  │          后端写入 teachers 表，同步写入 teacher_courses 关联表
  │             ↓
  │          前端关闭 Modal，刷新列表
  │
  ├─ 编辑教师 → 点击"编辑"按钮 → 弹出预填表单
  │             修改字段 → 提交 PUT /api/teachers/{id}
  │             ↓
  │          后端更新 teachers 表，删除旧 teacher_courses 记录并重新插入
  │             ↓
  │          前端关闭 Modal，刷新列表
  │
  └─ 删除教师 → 点击"删除"按钮 → 弹出确认对话框
                确认 → DELETE /api/teachers/{id}
                ↓
             后端将 deleted 字段置为 1（逻辑删除）
                ↓
             前端从列表移除该条目

学生、课程、教室、班级管理流程与上述一致，均为分页列表 + 新增/编辑 Modal + 逻辑删除。
班级管理额外包含学生成员管理：编辑班级时可选择加入或移出 class_students 关联记录。
```

## 六、异常处理流程

```plain
任意接口请求过程中：
  ↓
判断：请求是否成功？
  ├─ 网络错误（无响应）→ Axios 超时或 fail → 显示"网络错误，请检查连接" → 停留当前页
  │
  ├─ 401 未授权 → 清空 localStorage 中的 token 和 userInfo
  │               → Pinia Store 重置
  │               → 跳转登录页
  │               → 显示"登录已过期，请重新登录"
  │
  ├─ 403 无权限 → 显示"权限不足" → 停留当前页
  │
  └─ 业务错误（code ≠ 200）→ Axios 响应拦截器抛出 Error(message)
                             → 各页面 catch 块捕获 → 显示后端返回的错误描述
```

## 七、权限控制流程

```plain
用户发起任意请求
  ↓
前端 request.ts 请求拦截器：
  从 localStorage 读取 token → 附加到 Authorization 请求头
  ↓
后端 JwtAuthFilter（每次请求执行）：
  1. 提取请求头中的 Bearer Token
  2. 调用 JwtUtil.parseToken() 解析并验证签名
  3. 判断：Token 是否有效？
     ├─ 否 → 不注入身份信息 → Spring Security 拦截 → 返回 401
     └─ 是 → 从 Token 中取出 userId → 查询数据库获取完整 User 对象
              → 注入到 SecurityContext
  ↓
Spring Security 按 SecurityConfig 中的 URL 规则匹配：
  ├─ 路径需要 ADMIN 角色 + 当前用户是 teacher → 返回 403
  ├─ 路径需要已登录 + token 有效 → 放行进入 Controller
  └─ 公开路径（/api/auth/login）→ 直接放行
  ↓
Controller 通过 @AuthenticationPrincipal User user 注入当前用户
  ↓
Service 层按需使用 user.getRole() / user.getTeacherId() 进行业务级数据隔离
```

## 八、数据流转示意

```plain
┌─────────────────┐
│    用户浏览器    │
│  (Vue 3 SPA)    │
└────────┬────────┘
         │ ① 用户操作（点击、填写表单）
         ↓
┌─────────────────┐
│   前端逻辑层    │
│  Pinia Store    │
│  Vue Router     │
│  Axios 请求层   │
└────────┬────────┘
         │ ② HTTP 请求（携带 JWT Token）
         ↓
┌─────────────────┐
│   后端 API 层   │
│  Spring Boot 3  │
│  Spring Security│
│  Controller     │
│  Service        │
└────────┬────────┘
         │ ③ SQL 查询 / 写入
         ↓
┌─────────────────┐
│    数据库层     │
│    MySQL 8      │
│  MyBatis-Plus   │
└────────┬────────┘
         │ ④ 返回数据
         ↓
┌─────────────────┐
│   后端 API 层   │
└────────┬────────┘
         │ ⑤ JSON 响应 { code: 200, data: {...} }
         ↓
┌─────────────────┐
│   前端逻辑层    │
│ 响应拦截器解包  │
└────────┬────────┘
         │ ⑥ 渲染界面 / 更新状态
         ↓
┌─────────────────┐
│    用户浏览器    │
└─────────────────┘
```

## 九、流程图绘制建议

### 9.1 主流程图（泳道图）

建议使用泳道图（Swimlane Diagram），分为 3 个泳道：

| 泳道 | 内容 |
|------|------|
| **用户操作层** | 点击按钮、填写表单、查看结果 |
| **前端处理层** | 页面跳转、数据校验、Pinia 状态更新、API 调用 |
| **后端服务层** | 接口处理、权限验证、数据库操作 |

### 9.2 关键节点标注

| 图形 | 含义 | 示例 |
|------|------|------|
| **菱形** | 判断节点 | 是否登录？权限是否通过？ |
| **矩形** | 处理节点 | 生成 Token、写入数据库、计算课表 |
| **圆角矩形** | 页面/界面节点 | 登录页、仪表盘、周课表 |
| **圆柱体** | 数据库操作 | 查询、插入、逻辑删除 |
| **箭头** | 流程方向 | 实线箭头表示主流程，虚线箭头表示异常流程 |

### 9.3 建议分图

由于流程较复杂，建议拆分为 **4 张独立流程图**：

#### 图 1：用户认证流程图
- **范围**：登录 + JWT 认证 + 权限拦截
- **关键节点**：BCrypt 验证、Token 生成、SecurityContext 注入
- **适用章节**：论文"用户认证模块"

#### 图 2：排课管理流程图
- **范围**：手动排课 + 四维课表查询
- **关键节点**：按周生成记录、CSS Grid 渲染、日期对齐周一
- **适用章节**：论文"排课管理模块"

#### 图 3：调课申请审批流程图
- **范围**：教师发起 → 管理员审批 → 通过/拒绝/撤销
- **关键节点**：班级归属校验、状态流转（待审核→已通过/已拒绝）
- **适用章节**：论文"调课管理模块"

#### 图 4：基础数据管理流程图
- **范围**：教师/学生/课程/教室/班级 CRUD
- **关键节点**：逻辑删除、关联表维护
- **适用章节**：论文"基础数据管理模块"
